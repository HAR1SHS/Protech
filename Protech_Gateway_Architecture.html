Gateway / PortApps — The Edge Ingestion Layer

Protech's Building Management System (BMS) telemetry platform. Field devices in McDonald's stores stream $…* ASCII frames over raw TCP. The PortApps tier receives, validates, decodes and persists them into SQL Server — the upstream source that ultimately feeds the Databricks Medallion pipeline.

Protocol: raw TCP + ASCII frames

1What PortApps Is

The deployment/runtime tier of the BMS platform. Six responsibilities, in order:

1 · Listen

Bind a range of raw TCP ports for incoming device connections.

2 · Receive & Parse

Read the $…* ASCII data frames off the socket.

3 · Validate

Verify each frame with the Protech protocol checksum.

4 · Resolve

Look up site / customer from a cached master table.

5 · Persist

Write readings to SQL Server — stored procs (legacy) or staging + bulk copy (new).

6 · Act

ACK the device, control HVAC switches, surface health & anomalies on a live dashboard.

3The Components

New · Production Core

BmsSmart-DataGateway

The heart of the system (~1 MB, .NET 8 Windows Service). Replaces all DP/LP apps by listening on ports 31001–31009 at once. Extremely active — Serilog files reach 200 MB–1 GB/day.

  • TcpGatewayService — accept, ACK device, enqueue packet
  • PacketQueue — bounded queue with back-pressure
  • SensorDataHandler — decode 00D, SqlBulkCopy → staging
  • StagingFlushService — bulk copy ≤50k rows → final tables
  • DashboardService — embedded "Gateway Cockpit" UI
  • AI/anomaly — z-score, error-rate, EWMA, episodes
New · HVAC Control

BmsSmart-CtrlGateway

Smaller (94 KB, .NET 8 Windows Service), low-traffic. Deliberately independent from the DataGateway so HVAC control survives a data-pipeline restart. Ports 16002 / 16005.

  • SwitchControlService — drive HVAC relays on a schedule; auto-restart listener after socket error
  • CtrlStatusService — http://localhost:5101/api/status
  • CtrlHealthWriterService — monitor table every 30s
  • SwitchHealthTracker — per-switch health & ordering
New · Library

BmsSmart.Shared

Shared by both gateways (and BuildSmart.Api). Key piece: SelfEncryptingConfig — on first run encrypts appsettings.json → appsettings.enc via Windows DPAPI and deletes the plaintext. Also handles timezone normalization.

Legacy · Per-Port

DP / LP Port Processors

Same compiled program, different config — one console app owns one TCP port. Flow: accept TCP → checksum (CheckSum.dll) → parse (SharedLib.HeaderObj) → resolve site (24h cache) → EXEC stored proc → throttled gateway-status update.

  • DP-31002 → SD table
  • DP-31009 → SD table config bug: points at 31002 SP
  • LP-30009 → LV table

4The Data Protocol (Packet Format)

Devices send positional, fixed-width ASCII frames delimited by $ … * (no separators). The first 56 bytes are a fixed-width header; data blocks follow, then a 5-digit checksum and the * terminator. Customer_utilities.csv defines how to slice and decode every field. See the IoT Payload Visualizer tab for an interactive byte-by-byte decode.

FieldBytesExampleMeaning
SOF1$Start-of-frame delimiter
Version3006Protocol version (“006” = v6)
No. of Bytes3182Total string length in bytes (varies by meter type)
Port Code300DRouting — 00D → Data Port → SD (Stored Data) · 00T → Data Port → LD (Live Data) · 00Q → Control Port
Customer ID10McDonalds1Fixed-width site / customer identifier (resolved via master table)
Station ID102017000023Unique gateway / communication device ID
Meter Type40002Equipment category — 0002 Electricity, 0003 Temperature, 0005 LPG, 0011 Running Hours, 0030 Load Meter
Meter Serial40001Logical meter index at the site (sequential per type)
Date1005/03/2026Transaction date — DD/MM/YYYY
Time810:59:16Transaction time — HH:MM:SS (immediately follows date, no space)
Header total: 1 + 3 + 3 + 3 + 10 + 10 + 4 + 4 + 10 + 8 = 56 bytes
Data Blocksvariable468F1C85…Readings — IEEE-754 Float32 (8 hex = 4 bytes) for Electricity; 10-digit decimal for Load Meters; ASCII signed decimal for Temperature & LPG
Checksum509645Validation value — last field before the EOF delimiter
EOF1*End-of-frame delimiter
Decode rule: each catalog row gives a Byteposition, TotalBytes, DataType (HEX = IEEE-754 big-endian float / ASC = ASCII number / TXT = text) and a Multiplier, so value = decode(frame[pos : pos+len]) × Multiplier.

Why staging + bulk copy: packet rate is very high (logs hit GB/day). Writing to staging and bulk-copying 50k-row batches into final tables scales far better than legacy per-row stored-procedure inserts.
