LoRaWAN Deep Dive: Gateway & Communication

An in-depth look at how ProTech field modules securely communicate with the backend infrastructure.

📡 What is a LoRaWAN Gateway?

In the ProTech architecture, a LoRaWAN Gateway acts as a transparent, "dumb" bridge between the physical radio frequency (RF) world of the sensors and the IP/Internet world of the backend databases.

How it works: The gateway contains an antenna that listens across multiple radio channels simultaneously. When a field module (like an HVAC or LPG sensor) broadcasts a packet over the air using LoRa modulation, the gateway picks it up. It does not decrypt the data, process the payload, or evaluate rules. Instead, it wraps the raw RF packet, adds metadata (like signal strength, SNR, and timestamp), and forwards it via a standard IP connection (Ethernet/Wi-Fi/Cellular) to the Network Server.

🔐 Security & Cryptographic Keys

LoRaWAN uses a highly secure dual-key architecture based on AES-128 encryption. This ensures that the telecom provider managing the gateways and network cannot read the actual sensor data. Security is applied both at the network routing layer and the application data layer.

Network Session Key

NwkSKey

Shared between the end-device and the Network Server. It is used to calculate and verify the Message Integrity Code (MIC) to ensure the packet hasn't been tampered with. It also encrypts internal MAC (network management) commands.

Application Session Key

AppSKey

Shared exclusively between the end-device and the Application Server (your Data Port). This key is used for end-to-end encryption of the actual sensor payload (e.g., power consumption, temperature). The gateway and network server cannot decrypt this data.

Device Activation: Devices usually join the network using Over-The-Air Activation (OTAA), where they negotiate fresh Session Keys dynamically using a root AppKey, ensuring high security and preventing replay attacks.

📦 LoRaWAN MAC Packet Structure

Every radio transmission generated by the field modules follows a strict MAC (Medium Access Control) frame format before the raw data is wrapped inside the ProTech specific 56-byte header payload. The LoRaWAN MAC structure handles the routing, encryption overhead, and integrity checks.

Field Name Size Description
MHDR (MAC Header) 1 Byte Specifies the message type (e.g., Join Request, Confirmed Uplink, Unconfirmed Downlink) and the protocol major version.
DevAddr (Device Address) 4 Bytes A 32-bit identifier assigned to the device after it joins the network. Used by the server to identify which sensor sent the data.
FCtrl (Frame Control) 1 Byte Contains network control bits, such as the ADR (Adaptive Data Rate) bit, allowing the network to tell the device to lower its transmit power to save battery.
FCnt (Frame Counter) 2 Bytes A sequentially incrementing counter for every transmission. The server tracks this to detect and drop duplicate packets or replay attacks.
FPort (Port) 1 Byte Indicates the application port. Port 0 is reserved for MAC commands, while Ports 1-223 are used for standard application data.
FRMPayload (Encrypted Payload) Variable This is the actual sensor data (which includes your 56-byte ProTech header). It is fully encrypted using the AppSKey using AES-128.
MIC (Message Integrity Code) 4 Bytes A cryptographic checksum calculated over the entire message using the NwkSKey. The server recalculates this to verify the message is authentic and unaltered.
